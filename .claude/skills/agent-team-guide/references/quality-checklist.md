# Agent Teams スキル品質チェックリスト

チームスキルのレビュー・改善に使う。
全項目を既存スキルに適用し、未達の項目を改善する。

## 使い方

1. 対象スキルの SKILL.md と agents/*.md を Read する
2. 以下のチェック項目を順に確認
3. `[ ]` を `[x]` に変えながら進める
4. 未達の項目を改善点としてリストアップ
5. 改善を適用したら再チェック

---

## 1. SKILL.md チェック

- [ ] ヒアリング項目が具体的（「何を聞くか」が明確）
- [ ] エージェント一覧テーブルがある（name, 役割, 主な手段）
- [ ] タスク依存関係が明記されている（addBlockedBy の具体値）
- [ ] フィードバックループの Round 1/2/3 が定義されている
- [ ] 緊急通知パターンが定義されている（各エージェントがどんな問題を即時共有するか）
- [ ] 最終レポートのテンプレートへのリンクがある
- [ ] クリーンアップ手順（shutdown_request + TeamDelete）がある
- [ ] 起動設定が記載されている（subagent_type, team_name, mode, run_in_background）

## 2. エージェント定義チェック

- [ ] YAML frontmatter に name, description, tools, model がある
- [ ] 3-Phase 構造（初期化 → アクティブ → 最終化）
- [ ] Phase 1 に TaskList → TaskGet → TaskUpdate の手順
- [ ] Phase 3 に TaskUpdate(completed) + team-lead への SendMessage 手順
- [ ] SendMessage の名指しテンプレートがある（→ {agent-name}: {具体的依頼}）
- [ ] **通信効率ルール**が「コミュニケーションルール」セクションにある（ACK不要 + 待ちすぎない）
- [ ] 評価基準・判断基準が定義されている（何が「良い」か明確）
- [ ] 早期フィードバックの指示がある（blockedBy 待機中でも他のエージェントにフィードバック可能）

## 3. 通信設計チェック

- [ ] 全エージェントが少なくとも1つの SendMessage パターンを持つ
- [ ] 名指し依頼（→ agent-name: 具体的アクション）がある
- [ ] 緊急通知パターンがある（重大問題の即時共有）
- [ ] フィードバック受信後の修正→再共有フローがある
- [ ] broadcast は全員共有が必要な場面のみに限定されている
- [ ] summary フィールドの例がある（5-10語で具体的に）

## 4. タスク依存チェック

- [ ] 並列可能なタスクの依存が不必要に直列化されていない
- [ ] 全タスクに owner が設定される手順がある
- [ ] TaskCreate に description と activeForm が含まれている
- [ ] 最終タスク（レポート作成）が全タスク完了後に実行される
- [ ] 早期フィードバックの仕組みがある（依存待ちでもレビュー可能）

## 5. プロンプト品質チェック

- [ ] プロンプトにプロダクト情報が含まれる手順がある（省略厳禁）
- [ ] プロンプトにコンテキストファイルパスが含まれる
- [ ] プロンプトにチーム構成（他エージェント名と役割）が含まれる
- [ ] 既存の context.md がある場合、その内容をプロンプトに含める手順がある

## 6. コンテキスト管理チェック

- [ ] ファイル構成が定義されている
- [ ] context-manager（または同等の役割）がいる
- [ ] 議事録（log.md）が時系列で更新される
- [ ] 前回セッションからの引き継ぎが可能
- [ ] 大量データはファイルに書き出し、パスだけ SendMessage で共有する設計

## 7. MCP ツール統合チェック（該当する場合のみ）

- [ ] MCP を使うエージェントは `subagent_type: "general-purpose"` で定義
- [ ] agents/*.md ではなくプロンプト内で役割を定義している
- [ ] MCP ツール名がプロンプト内に明記されている
- [ ] 複数エージェントが同じファイルに書く場合、競合防止策がある（フレーム分離等）

---

## よくある改善ポイント

| 問題 | 原因 | 対策 |
|------|------|------|
| エージェントが暇すぎる | 依存関係が過剰 | 並列可能なタスクの依存を外す + 早期フィードバック設計 |
| 通信が一方通行 | 名指し依頼がない | `→ agent-name: 具体的アクション` パターンを追加 |
| ACK だけの返信が大量発生 | 旧「無視禁止」ルールの副作用 | 「実質的な内容のある返信のみ」ルールに変更 |
| エージェントが全員の返信を待ちすぎ | 「待ちすぎない」ルールがない | 2人以上の反応で次フェーズへ進む設計に変更 |
| 最終成果物の品質が低い | レビュアーがいない | レビュアー/チェッカー役を追加 |
| デバッグが困難 | ログがない | PostToolUse Hook (`agent-teams-log` プラグイン) を有効化 |
| セッション間で情報が失われる | コンテキストファイルがない | context-manager + init.sh を追加 |
| MCP ツールが使えない | agents/*.md で定義している | `subagent_type: "general-purpose"` に変更 |
| プロダクト情報が伝わらない | プロンプトで省略している | 全情報をプロンプトに展開（省略厳禁） |
| タスクの目的が不明確 | description が空 | 成果物・アクション・ファイルパスを description に含める |
