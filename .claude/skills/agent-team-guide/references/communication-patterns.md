# 通信パターン

## 4つの通信モデル

### 1. Broadcast モデル（全員共有型）

```
specialist-A → 全員に調査結果を共有
全員 → specialist-A にフィードバック
specialist-A → 修正版を全員に再共有
```

**使う場面**: 同一対象を複数観点で評価する（app-naming, ui-review）
**利点**: 全員が全情報を持つ、見落とし防止
**欠点**: メッセージ量が多い（N人 × N人）、コスト高

### 2. Directed モデル（指名型）

```
architect → researcher に調査依頼
researcher → architect に結果報告
architect → writer に執筆指示
writer → reviewer にレビュー依頼
```

**使う場面**: 作業が段階的に進む（skill-creator-team）
**利点**: 通信量が少ない、依存関係が明確
**欠点**: ボトルネックが発生しやすい

### 3. Hybrid モデル（指名+選択的共有）

```
architect → 全員にセグメント設計を共有（broadcast）
bias-reviewer → architect に早期フィードバック（directed）
researcher → architect にデータ提供（directed）
architect → writer に執筆指示（directed）
writer → bias-reviewer にレビュー依頼（directed）
```

**使う場面**: 段階的進行だが一部の成果物は全員に見せたい（persona-creation, aso-optimize）
**利点**: 必要な情報だけが全員に届く、コスト効率が良い
**欠点**: 設計がやや複雑

### 4. Subteam モデル（サブチーム型）

```
Team-1: designer-1 → reviewer-1 + copywriter-1
Team-2: designer-2 → reviewer-2 + copywriter-2
（チーム間通信なし）
```

**使う場面**: 並列で独立した成果物を作る（ui-variations）
**利点**: 最大並列度、チーム間干渉なし
**欠点**: エージェント数が多い（N × チームサイズ）

### モデル選択ガイド

| 質問 | → モデル |
|------|---------|
| 全員が同じ対象を見る？ | Broadcast |
| 作業が A→B→C と段階的？ | Directed |
| 段階的だが初期成果は全員に見せたい？ | **Hybrid（推奨）** |
| 独立した成果物を並列で作る？ | Subteam |

## SendMessage の実際の構文

### type: "message"（特定の相手に送信）

```
SendMessage(
  type: "message",
  recipient: "researcher",      # 相手のエージェント名
  content: "調査結果を共有...",  # 本文
  summary: "市場リサーチ結果共有" # 5-10語の要約（UI プレビュー用）
)
```

### type: "broadcast"（全員に送信）

```
SendMessage(
  type: "broadcast",
  content: "全員に共有します...",
  summary: "セグメント設計完了"
)
```

**broadcast の注意**: N人分の個別配信が走る。コストが高い。以下の場合のみ使う:
- 全員が同じ情報を必要とする初期共有
- 緊急通知（問題発見時）

通常のやりとりは `type: "message"` で名指し送信する。

### summary フィールド

5-10語の短い要約。UI のプレビューに表示される。本文と異なる内容にしない。

**良い例**: `"セグメント設計4案を共有"`, `"ペルソナ01のバイアス指摘"`
**悪い例**: `"メッセージ"`, `"結果"`, `"確認お願いします"`

## 通信効率ルール

**全エージェント定義の「コミュニケーションルール」セクションに必ず含める**:

```markdown
## コミュニケーションルール
- **「了解しました」だけのACK返信は不要** - フィードバック・質問・修正結果など、相手の次のアクションに繋がる内容がある場合のみ返信する
- **待ちすぎない**: フィードバックを全員から集める必要はない。2人以上から反応があれば次のフェーズに進む。5分以上フィードバックがなければ先に進んでよい
- {役割固有のルール}
```

### なぜ「ACK不要」か

旧ルール（無視禁止）では「受領しました」だけの返信が大量に発生していた:
- 5人チームで1通の broadcast → 4通の「受領しました」→ 計20通以上の無意味なやりとり
- トークン消費が増大し、コンテキストウィンドウを圧迫
- エージェントが返信待ちで本来の作業に着手できない

**新ルール**: 返信すべき時 vs しなくてよい時

| 返信すべき | 返信不要 |
|-----------|---------|
| フィードバック・修正指摘がある | 「了解しました」「ありがとう」だけ |
| 質問がある | 内容に異論がなく次のアクションもない |
| 修正完了の報告 | 情報を受け取っただけで作業が変わらない |
| 緊急の問題発見 | 単なる進捗確認への応答 |

### なぜ「待ちすぎない」か

旧パターンでは全員のフィードバックを待ってから次に進んでいた:
- 5人中1人の返信が遅いだけでチーム全体が停滞
- 全員一致を求めると永遠に次のフェーズに進めない

**新ルール**: 2人以上から反応があれば次へ進む。残りのフィードバックは Phase 2 で対応できる。

## 名指し依頼パターン

効果的な通信の鍵は **名指しで具体的に依頼する** こと。

**悪い例**（名指しなし、曖昧）:
```
候補リストを作りました。フィードバックお願いします。
```

**良い例**（名指し、具体的）:
```
→ legal-researcher: 各候補の商標リスクを調査してください。「使用不可」があれば即座に教えてください。
→ digital-presence: 各候補のSEO/ASO/SNS取得可能性を評価してください。
→ global-checker: 多言語でのネガティブな意味や発音問題をチェックしてください。
→ context-manager: candidates.md に候補リストを記録してください。
```

**ポイント**:
- `→ {agent-name}:` で始める（誰への依頼か一目でわかる）
- 動詞で終わる（「調査してください」「記録してください」）
- 期待するアウトプットを明示する（「使用不可があれば即座に教えて」）

## 緊急通知パターン

問題発見時に即座に全員に警告する:

```
SendMessage(
  type: "broadcast",
  content: "[全員へ] ⚠️ {問題の概要}
理由: {具体的理由}
→ {対応すべきエージェント}: {具体的なアクション}
→ context-manager: この問題をログに記録してください。",
  summary: "緊急: {問題の簡潔な説明}"
)
```

**緊急通知すべき事例**:
- 候補/成果物が使用不可と判明
- セグメント仮説を否定するデータを発見
- 重大なバイアス・差別的表現を検出
- ファイル書き込みの競合を検知

## リーダーの通信ルール

1. **見守りが基本**: エージェントが自律的に議論する間は介入しない
2. **介入タイミング**:
   - エージェントが同じメッセージを繰り返している
   - 矛盾が 2 往復以上解消されない
   - エージェントが blockedBy 待ちのまま進まない
   - 明らかに方向性がずれている
3. **介入方法**: `SendMessage(type: "message", recipient: "{agent-name}")` で個別に指示
4. **最終統合**: 全エージェントの報告を統合してレポート作成
5. **シャットダウン**: 全員に `shutdown_request` → 承認後 `TeamDelete`

## メッセージの長さ

- **短いメッセージ**: ACK だけの返信は不要。フィードバックや質問がある場合のみ 1-3 行で返す
- **中程度のメッセージ**: 調査結果やフィードバックは要点を絞る
- **長いメッセージ**: 詳細データは「ファイルに書き出して、パスを共有する」方が効率的

```
→ context-manager: research-data.md に全データを記録してください。
→ persona-architect: research-data.md を参照し、セグメント設計に反映してください。
```

大量のデータを SendMessage 本文に含めるとトークンコストが高い。
context-manager にファイル化を依頼し、パスだけ共有する方が効率的。
