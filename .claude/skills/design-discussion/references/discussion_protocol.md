# 議論プロトコル

各ステップで実行する「議論ラウンド」の詳細。

---

## チームライフサイクル（CRITICAL）

Agent Teams で SendMessage を使うには、**事前に TeamCreate でチームを作成**する必要がある。

### セッション開始時: チーム作成

```python
# 1. 既存チームがあれば削除（前セッションの残りを掃除）
TeamDelete()  # エラーが出ても無視

# 2. 新しいチームを作成
TeamCreate(
  team_name="design-discussion",
  description="<設計テーマ>の議論チーム"
)
```

### 各ステップ: エージェント起動

Task の必須パラメータ:
| パラメータ | 値 | 説明 |
|-----------|-----|------|
| `team_name` | `"design-discussion"` | TeamCreate で作成したチーム名 |
| `name` | エージェント名 | SendMessage のルーティングに使用 |
| `subagent_type` | `"design-discussion:<agent-name>"` | エージェント定義の指定 |
| `mode` | `"bypassPermissions"` | 権限バイパス |
| `run_in_background` | `true` | バックグラウンド実行 |

起動テンプレートの詳細は `references/agent_spawn_templates.md` を参照。

### セッション終了時: チーム削除

```python
TeamDelete()
```

## ラウンド 1: チーム議論（R1）

ステップに関係するエージェントを**Agent Teams として起動**し、
**最初から SendMessage で相互にやり取りしながら**提案・反論・合意形成を行う。

**議論の進め方**:
1. 各エージェントが自分の視点から初期分析を SendMessage で共有
2. 他のエージェントの分析に対して反論・補足・質問を SendMessage で返す
3. 対立点が残る場合は議論を継続し、収束を目指す
4. ユーザーに確認が必要な場合は user-liaison に質問リクエストを送る
5. 合意点・対立点をまとめて議論を終了

**収束条件（往復数制限なし）**:
- 全エージェントが合意に達した（新たな反論が出なくなった）
- 対立点が明確に整理され、ユーザー判断に委ねることが合意された
- 議論が同じ論点の繰り返しになった（収束のシグナル）

**ユーザーへの質問（CRITICAL）**:
エージェントは直接 `AskUserQuestion` は使わない。
必ず user-liaison に SendMessage で質問リクエストを送る。

```
[ASK_USER_REQUEST]
質問: [ユーザーに聞きたいこと]
背景: [なぜこの情報が必要か]
緊急度: [高/中/低]
依頼元: [エージェント名]
```

## ラウンド 2: スコアリング評価（R2）

チーム議論の中でスコアリングも実施する。
議論の終盤で各エージェントが評価を出し合い、SendMessage で合意する。

**スコアリング基準（5軸、各 1-5 で評価）**:
| 評価軸 | 説明 | 主な評価者 |
|--------|------|----------|
| 技術的実現性 | 現在のスタックで実装可能か | engineer |
| 工数・コスト | 初期実装 + 保守の総合コスト | engineer |
| 保守性・拡張性 | 将来の変更容易性 | solution-architect |
| ビジネス価値 | ユーザー・事業への影響度 | product-manager |
| リスク（反転） | リスクが低いほど高スコア | devils-advocate |

**スコアリング出力フォーマット**:
```
| 評価軸       | 案A | 案B | 案C |
|------------|-----|-----|-----|
| 技術的実現性  | 4   | 3   | 5   |
| 工数・コスト  | 5   | 3   | 4   |
| 保守性・拡張性| 3   | 5   | 3   |
| ビジネス価値  | 4   | 4   | 3   |
| リスク       | 2   | 4   | 3   |
| 合計         | 18  | 19  | 18  |
```

スコアが低い項目には必ず理由を付記する。

---

## ラウンド 3: Devil's Advocate（R3）

チーム議論の後、devils-advocate が専任で合意案を批判する。

devils-advocate へのプロンプトに含める情報:
```
以下はチームの現時点での合意案です。
この案の致命的な欠陥・見落とし・隠れたリスクを全力で見つけてください。

現時点の合意案:
[合意内容]

スコアリング結果:
[スコアリング表]
```

**Devil's Advocate の結果処理**:
1. 致命的な指摘（全員が認めた問題）→ 合意案を修正
2. 重要だが対処可能な指摘 → リスクとして記録、対策を追記
3. 過剰な指摘（他のエージェントが反論できた）→ 却下理由を記録

---

## ラウンド 4: 統合・ユーザー提示（R4）

Facilitator（SKILL.md）が:
1. 合意点をまとめる
2. 対立点を明示する
3. スコアリング結果を提示する
4. Devil's Advocate で発見された懸念を提示する
5. 未解決の対立点についてユーザーに判断を求める

**ユーザーへの提示フォーマット**:
```
━━━━ チーム議論結果 ━━━━

スコアリング:
  | 評価軸       | 案A | 案B | 案C |
  |------------|-----|-----|-----|
  | 技術的実現性  | 4   | 3   | 5   |
  | 工数・コスト  | 5   | 3   | 4   |
  | 保守性・拡張性| 3   | 5   | 3   |
  | ビジネス価値  | 4   | 4   | 3   |
  | リスク       | 2   | 4   | 3   |
  | 合計         | 18  | 19  | 18  |

合意:
  - [全員が同意した点]

対立（ユーザー判断が必要）:
  1. [論点A]
     - solution-architect: 「〇〇すべき」（理由: ...）
     - engineer: 「△△のリスクがある」（理由: ...）

Devil's Advocate 懸念:
  - 致命的: [あれば]
  - 注意点: [一覧]

どう判断しますか？
```

## ユーザー判断の追跡

対立点をユーザーが判断した場合、**判断内容と理由を必ず記録する**。

### 記録フォーマット

Facilitator はユーザーの判断を受けたら、必ず以下を `discussion_log.md` に追記する:

```markdown
## ユーザー判断 - YYYY-MM-DD HH:MM

### 判断内容
- 採用: [採用された内容]
- 却下: [却下された内容]

### 対立の背景
- [Agent A]: 「[主張]」（理由: [...]）
- [Agent B]: 「[主張]」（理由: [...]）

### ユーザーの判断理由
> [ユーザーの発言から判断理由を抽出。不明な場合は質問する]

### 影響範囲
- [この判断が影響するステップ/成果物]

---
```

### 理由が不明な場合

ユーザーが理由を述べずに判断した場合、Facilitator は確認する:

```
その判断の理由を教えていただけますか？
後で振り返る時に「なぜこう決めたか」が分かると助かります。
```

## ディスカッション議事録（CRITICAL）

**ファイル書き込みの責務**:
各エージェントは SendMessage で議論結果・分析・合意点を Facilitator に報告する。
Facilitator（SKILL.md）が Write ツールで各 `.md` ファイルに書き込む責任を持つ。
エージェント自身はファイルを直接書き込まない（エージェント定義に Write/Edit ツール不要）。

各ラウンドの結果を `discussion_log.md` に追記する。

**議事録の記録フォーマット**:
```markdown
## Step [N] Round [R]: [ラウンド名] - YYYY-MM-DD HH:MM

### [エージェント名] → [エージェント名] への議論
> [発言内容]

### [エージェント名] → [エージェント名] への反論
> [反論内容]

### 合意点
- [合意内容]

### 対立点（未解決）
- [論点]: [各エージェントの立場]

---
```

**ディレクトリ構造**:
```
.claude/design_discussion/sessions/<discussion-slug>/
├── session.json          # セッション進捗管理
├── discussion_log.md     # ディスカッション議事録
├── 1_clarification.md   # Step 1の結果
├── 2_candidates.md      # Step 2の結果
├── 3_comparison.md      # Step 3の結果
└── 4_decision.md        # Step 4の結果
```
