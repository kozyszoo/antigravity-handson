# 品質管理: Quality Gate & ステップ間整合性チェック

---

## Quality Gate

各ステップの完了時に、定量的な品質基準を満たしているかチェックする。
Quality Gate を通過しないステップは完了とせず、追加議論またはユーザー判断を求める。

### 全ステップ共通基準

| 基準 | 条件 | 未達時のアクション |
|------|------|----------------|
| スコアリング最低点 | 全評価軸 ≥ 3 | スコア2以下の軸について追加議論 |
| 未解決の対立 | 0件 | ユーザーに対立点の判断を求める |
| Devil's Advocate 致命的指摘 | 0件（対処済み） | 合意案を修正して再評価 |
| ASK_USER 未回答 | 0件 | ユーザーに回答を求める |

### ステップ固有基準

**Step 1: 課題深掘り**
| 基準 | 条件 |
|------|------|
| 課題の明確化 | 「誰の」「何の」課題か一文で言語化できる |
| フォッグモデル | 動機・能力・きっかけの3つが特定されている |
| ペルソナ整合性 | UX Analyst がペルソナとの整合性を確認済み |

**Step 2: 代替案**
| 基準 | 条件 |
|------|------|
| 代替案数 | 3案以上 |
| コードベース調査 | Engineer が既存実装を調査済み |
| UI視覚化 | Designer が全案を ASCII art で視覚化済み |
| 行動分析 | BP が全案の習慣ループを評価済み |

**Step 3: スコープ**
| 基準 | 条件 |
|------|------|
| MVP定義 | MVP/Nice to Have/Future に全項目分類済み |
| 習慣ループ完備 | MVPにきっかけ→行動→報酬が含まれる（BP確認） |
| 技術的整合性 | Engineer がMVP工数を見積もり済み |

**Step 4: 要件定義**
| 基準 | 条件 |
|------|------|
| ユーザーストーリー | 全MVP項目にストーリーがある |
| 受け入れ条件 | 全ストーリーに Given/When/Then がある |
| 全員レビュー | 全エージェントがレビュー完了 |

**Step 5: UI設計**
| 基準 | 条件 |
|------|------|
| 画面網羅 | 全MVP画面のレイアウトがある |
| PC/スマホ対応 | 両方のレイアウトがある |
| 認知負荷チェック | BP が認知負荷を評価済み |
| 既存コンポーネント | Engineer が再利用可能なコンポーネントを特定済み |

### Quality Gate 判定フォーマット

```
━━━━ Quality Gate チェック ━━━━

Step <N>: <ステップ名>

  ✅ スコアリング最低点: 全軸 3 以上
  ✅ 未解決の対立: 0件
  ⚠️ Devil's Advocate: 致命的指摘 1件 → 修正中
  ✅ ASK_USER: 未回答 0件

  ✅ [ステップ固有基準1]: 達成
  ✅ [ステップ固有基準2]: 達成
  ❌ [ステップ固有基準3]: 未達 → [必要なアクション]

判定: FAIL → [未達項目]を解決してから再チェック
```

## ステップ間の整合性チェック

各ステップの開始時に、前ステップの成果物との整合性を自動チェックする。
矛盾が検出された場合、エージェントに議論を戻すか、ユーザーに確認する。

### チェックマトリクス

```
Step 1 ──→ Step 2: 課題定義が代替案に反映されているか
  │          │
  │          ↓
  ├───────→ Step 3: 課題定義がスコープに反映されているか
  │          │       選定案がスコープのベースになっているか
  │          │
  │          ↓
  ├───────→ Step 4: 課題がストーリーで解決されるか
  │          │       スコープ外の項目が混入していないか
  │          │
  │          ↓
  └───────→ Step 5: 全ストーリーの画面があるか
                     受け入れ条件を画面で実現できるか
                     行動設計メモがUIに反映されているか
```

### 各ステップの整合性チェック項目

**Step 2 開始時**（Step 1 との整合性）:
| チェック項目 | 確認方法 |
|------------|---------|
| 課題定義の反映 | 全代替案が Step 1 の課題を解決する案になっているか |
| ペルソナの一貫性 | Step 1 で特定したペルソナが評価対象になっているか |
| フォッグモデルの反映 | Step 1 の動機・能力・きっかけが代替案で考慮されているか |

**Step 3 開始時**（Step 1-2 との整合性）:
| チェック項目 | 確認方法 |
|------------|---------|
| 選定案との一致 | スコープが Step 2 で選定した案をベースにしているか |
| 課題カバレッジ | MVP で Step 1 の課題が最低限解決されるか |
| 却下案の混入防止 | Step 2 で却下した案の要素が紛れ込んでいないか |

**Step 4 開始時**（Step 1-3 との整合性）:
| チェック項目 | 確認方法 |
|------------|---------|
| MVP網羅 | 全MVP項目にユーザーストーリーがあるか |
| スコープ外の排除 | Nice to Have / Future の項目が要件に含まれていないか |
| 課題→ストーリー | Step 1 の課題が少なくとも1つのストーリーで解決されるか |

**Step 5 開始時**（Step 1-4 との整合性）:
| チェック項目 | 確認方法 |
|------------|---------|
| ストーリー→画面 | 全ユーザーストーリーに対応する画面があるか |
| 受け入れ条件→UI | 全受け入れ条件を画面上で実現できるか |
| 行動設計→UI | BP の行動設計メモ（デフォルト値、フィードバック等）がUIに反映されているか |
| コンポーネント一致 | Engineer が特定した再利用コンポーネントが使われているか |

### 整合性チェック実行フォーマット

```
━━━━ 整合性チェック ━━━━

Step <N> 開始前チェック:

  ✅ 課題定義の反映: Step 1 の課題がストーリーに含まれている
  ✅ スコープ範囲: MVP項目のみ、スコープ外なし
  ❌ 習慣ループ: Step 3 で指摘された「報酬」がストーリーに未反映
     → 対応: US-3 に報酬フィードバックを追加する必要あり

[❌がある場合]
整合性の問題が見つかりました。修正してから進めますか？
```

### 矛盾検出時のアクション

| 矛盾の種類 | アクション |
|-----------|----------|
| 軽微（表現の不一致） | Facilitator が自動修正し、ユーザーに報告 |
| 中程度（項目の漏れ） | 該当エージェントに追加作業を指示 |
| 重大（方針の矛盾） | 前ステップに戻って再議論を提案 |
