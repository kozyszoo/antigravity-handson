# 外部リサーチ & スキル連携

---

## ドメインドキュメント連携

各ステップ開始時、`docs/domain/` 配下のドキュメントを読み込み、議論を既存仕様に接地させる。

1. `Glob`で `docs/domain/*.md` を検索
2. ユーザーが言及した画面・機能に関連するドキュメントを読み込む
3. 読み込んだ内容を全エージェントにコンテキストとして渡す
4. エージェントはドメインドキュメントの内容を前提として議論する

**NG**: ドキュメントに書いてある仕様をユーザーに質問する
**OK**: ドキュメントを前提に、その先の判断をユーザーに質問する

## 競合・業界リサーチ

Step 2（代替案）で `WebSearch` を使い、議論を外部知見で補強する。
内部知識だけに閉じず、競合アプリや業界ベストプラクティスを参照して議論の幅を広げる。

### リサーチタイミング

- **Step 2 開始時**（必須）: Engineer のコードベース調査と並列で実行
- **議論中**（任意）: エージェントが「業界ではどうしているか知りたい」と判断した場合

### リサーチ手順

1. **競合アプリの特定**: ユーザーの機能要望に関連する競合アプリを `WebSearch` で検索
   - 検索例: `"[機能名] app UI pattern"`, `"[機能名] best practice UX"`
2. **実装パターンの調査**: 競合がどう実装しているかを調査
   - 検索例: `"[アプリ名] [機能名] how it works"`
3. **業界ベストプラクティス**: UX/UI のベストプラクティスを調査
   - 検索例: `"[機能カテゴリ] UX best practice 2025"`, `"[UIパターン名] design pattern"`

### リサーチ結果の活用

```
# Engineer と並列でリサーチエージェントを起動
Task(
  description="Research: competitive analysis",
  subagent_type="general-purpose",
  prompt=`
    以下の機能について、競合アプリと業界ベストプラクティスを調査してください。

    機能: [機能概要]
    課題: [Step 1 の課題定義]

    調査項目:
    1. この機能を持つ競合アプリ（3つ以上）の実装方法
    2. UX/UIの業界ベストプラクティス
    3. よくある失敗パターン

    出力フォーマット:
    ### 競合アプリの実装例
    | アプリ名 | 実装方法 | 強み | 弱み |
    |---------|---------|------|------|

    ### 業界ベストプラクティス
    - [パターン名]: [説明]

    ### よくある失敗パターン
    - [パターン名]: [なぜ失敗するか]
  `
)
```

リサーチ結果は全エージェントにコンテキストとして渡し、代替案の提案・評価に活用する。

## 次スキル連携

各ステップの成果物は、他のスキルの入力として活用できる。
セッション完了時、またはステップ完了時に、次に使うべきスキルを提案する。

```
┌─────────────────────────────────────────────────────────────┐
│                  Feature Discussion                         │
│                                                             │
│  Step 1 ─── Step 2 ─── Step 3 ─── Step 4 ─── Step 5       │
│  課題深掘り   代替案     スコープ    要件定義    UI設計       │
│                                                             │
└──────┬──────────┬──────────┬──────────┬──────────┬──────────┘
       │          │          │          │          │
       ▼          │          │          ▼          ▼
  ┌─────────┐     │          │   ┌───────────┐  ┌────────────────┐
  │ /jtbd   │     │          │   │/team-plan │  │/frontend-design│
  │ /5-whys │     │          │   │ 実装計画   │  │ UI実装         │
  └─────────┘     │          │   └───────────┘  └────────────────┘
                  ▼          ▼
           ┌───────────┐  ┌───────────────┐
           │/devils-   │  │/pre-mortem    │
           │ advocate  │  │ リスク分析     │
           └───────────┘  └───────────────┘
```

**ステップごとの連携先**:

| 完了ステップ | 連携先スキル | 用途 |
|------------|------------|------|
| Step 1 完了 | `/jtbd` | ユーザーの本質的な動機をさらに深掘り |
| Step 1 完了 | `/five-whys` | 課題の根本原因を特定 |
| Step 2 完了 | `/devils-advocate` | 選定案の追加批判 |
| Step 3 完了 | `/pre-mortem` | MVPのリスク事前分析 |
| Step 4 完了 | `/team-plan` | 要件定義 → 実装計画を作成 |
| Step 5 完了 | `/frontend-design` | UI設計プロンプト → 実装 |
| Step 5 完了 | `/ui-variations` | 複数UIバリエーションを生成・比較 |

**提案フォーマット**:
```
━━━━ 次のアクション ━━━━

このステップの成果物を活かして、以下のスキルに進めます：

  📋 /team-plan    → 要件定義をもとに実装計画を作成
  🎨 /frontend-design → UI設計プロンプトで画面を実装

どれに進みますか？（または「次のステップ」で Step <N+1> へ）
```
