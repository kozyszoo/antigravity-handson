Antigravityの「Globalルール」と「Workspaceルール」の最大の違いは、**「すべてのプロジェクトに適用されるか」**、それとも**「特定のプロジェクト（や特定の状況）限定か」**という点です。

ソースに基づき、それぞれの特徴と使い分けのポイントを解説します。

### 1. Globalルール（全プロジェクト共通）
Antigravity全体の設定として、どのプロジェクトを開いても常に適用させたいルールを記述します,。

*   **特徴**: 一度設定すれば、新しいプロジェクトを作るたびに設定し直す必要がありません。
*   **おすすめの使い道**:
    *   **言語設定**: 「回答は常に日本語で行ってください」。
    *   **基本姿勢**: 「コードの説明は省略して、コードのみを出力して」「常に敬語を使って」など、AIの性格や基本的な振る舞いの指定。
*   **設定場所**: `Customizations` > `Rules` > `Global`。

### 2. Workspaceルール（プロジェクト固有・条件付き）
現在開いているプロジェクト（ワークスペース）内でのみ有効にしたいルールです。さらに、**「特定のファイル形式」や「AIの判断」で発動条件を細かく制御できる**のがGlobalルールにはない大きな特徴です,。

*   **特徴**: プロジェクトごとに異なる要件や、技術スタック（Python, JavaScriptなど）に応じたルールを設定できます。
*   **おすすめの使い道**:
    *   **特定の技術スタック用**: 「Pythonを書くときはType Hintを必ずつける」「JavaScriptでは`const`を優先する」など。
    *   **プロジェクト固有の規約**: 「このプロジェクトではファイル名はスネークケースにする」「テストコードは必ず`tests/`フォルダに入れる」など。
*   **高度な設定（Activation Mode）**,:
    Workspaceルールは、いつ適用するかを以下の4つから選べます。
    1.  **Is On（常時）**: そのプロジェクト内では常に適用（デフォルト）。
    2.  **Manual（手動）**: チャットでユーザーがメンション（指名）したときだけ適用。
    3.  **Model Decision（AI判断）**: Description（説明）に書かれた状況（例：「Pythonファイル修正時」）だとAIが判断したときに自動適用。
    4.  **Glob（ファイル指定）**: `*.py` のように拡張子を指定し、そのファイル形式を扱うときだけ適用。

### 使い分けの具体例

| ルールの内容 | 推奨設定 | 理由 |
| :--- | :--- | :--- |
| **「日本語で話して」** | **Global** | どのアプリを作る時も日本語で話してほしいため。 |
| **「Pythonのコード規約」** | **Workspace** (Glob: `*.py`) | Python以外のプロジェクト（Webサイト制作など）では邪魔になるため、特定のファイル形式のみに適用させる。 |
| **「Reactのコンポーネント設計」** | **Workspace** | このプロジェクトはReactで作るが、次は別のフレームワークかもしれないため、プロジェクト単位で管理する。 |
| **「セキュリティ重視の検証」** | **Workspace** (Manual) | 常に厳しいチェックをすると遅くなるため、重要な場面で手動で呼び出して適用する。 |

**まとめると：**
まずは**「Globalルール」**で「日本語化」などの基本設定を済ませ、個別のアプリ開発で特定の言語やフレームワークを使う段になったら、その都度**「Workspaceルール」**で技術的な細かい指示を追加していくのが最も効率的な運用方法です。
