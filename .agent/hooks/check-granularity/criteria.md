# タスク粒度 評価基準 詳細

## 評価マトリクス

| 評価項目 | ✅ 適切 | ⚠️ 要注意 | ❌ 大きすぎる |
|---------|--------|----------|-------------|
| アクション数 | 1-2個 | 3個 | 4個以上 |
| 影響レイヤー | 1レイヤー | 2レイヤー（密結合） | 3レイヤー以上 |
| 変更ファイル数 | 1-3ファイル | 4-5ファイル | 6ファイル以上 |
| 責務の数 | 1つ | 1つ（+付随的なもの） | 2つ以上の独立した責務 |
| description の長さ | 1-3行 | 4-5行 | 6行以上 |
| 完了条件の明確さ | 1つの明確な条件 | 2-3の関連する条件 | 曖昧または複数の独立した条件 |

## レイヤー定義（一般的なWebアプリケーション）

| レイヤー | キーワード |
|---------|----------|
| FE (Frontend) | component, hook, page, UI, 画面, 表示, コンポーネント, フック |
| BE (Backend) | API, endpoint, route, service, repository, エンドポイント |
| DB (Database) | schema, migration, テーブル, カラム, DB, マイグレーション, スキーマ |
| Shared | type, client, 型定義, 共通 |

## 大きすぎるタスクの検出パターン

### 1. 接続詞パターン

以下の接続詞が subject や description に含まれる場合、複数の責務が混在している可能性が高い:

- 「〜と〜」「〜および〜」
- 「〜し、さらに〜」「〜も〜」
- 「additionally」「and also」「as well as」

### 2. 動詞の複数パターン

description に異なる動詞が3つ以上ある場合:

- 「作成する」「追加する」「修正する」「更新する」
- 「implement」「add」「fix」「update」「refactor」

### 3. 曖昧スコープパターン

以下のフレーズは具体性が不足している:

- 「〜機能を実装」（何の機能？CRUD全部？）
- 「〜画面を作成」（どの部分？全体？）
- 「〜を改善」（何をどう？）
- 「〜を対応」（何を？）

### 4. レイヤー横断パターン

description に以下のキーワードが複数レイヤーにまたがって出現:

- FE: 「コンポーネント」「画面」「UI」「フック」「表示」
- BE: 「API」「エンドポイント」「サービス」「ルート」
- DB: 「スキーマ」「マイグレーション」「テーブル」「カラム」

## 分割時の粒度ガイドライン

### 理想的なタスクサイズ

- **subject**: 動詞1つ + 対象1つ（例: 「タスク作成モーダルにバリデーションを追加」）
- **description**: 具体的な実装内容を2-3行で記述
- **影響範囲**: 1-3ファイル
- **完了条件**: 「〜ができること」の形式で1つ

### 分割のやりすぎ注意

以下は分割しすぎ:

- ❌ 「import文を追加する」→ これ単体ではタスクにならない
- ❌ 「変数名をリネームする」→ 親タスクの一部として扱う
- ❌ 「テストを1つ追加する」→ 実装タスクに含める

### 適切な分割単位

- ✅ **1つの関数/フック**: useFocusQueue にソート機能を追加
- ✅ **1つのコンポーネント**: タスクカードにメモ表示を追加
- ✅ **1つのエンドポイント**: GET /api/tasks/:id を実装
- ✅ **1つのスキーマ変更 + マイグレーション**: tasks テーブルに priority カラムを追加
- ✅ **1つの画面の特定セクション**: 進捗ボードのヘッダー部分を実装
