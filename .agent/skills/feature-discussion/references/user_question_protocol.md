# ユーザーへの質問プロトコル（共通）

全エージェント共通の「ユーザーへの質問」ルール。
各エージェント定義ファイルから本ファイルを参照する。

---

## 基本ルール

議論中に「ユーザーに聞かないと判断できない」と感じたら、**`user-liaison` に SendMessage で質問リクエストを送る**。
直接 `AskUserQuestion` は使わない。user-liaison が質問の要否を判定し、統合・整理してユーザーに質問する。

## 質問リクエストのフォーマット

```
[ASK_USER_REQUEST]
質問: [ユーザーに聞きたいこと]
背景: [なぜこの情報が必要か]
緊急度: [高/中/低]
依頼元: [あなたのエージェント名]
```

## 緊急度の基準

| 緊急度 | 基準 | 例 |
|--------|------|-----|
| **高** | 議論がブロックされている。この情報なしでは前に進めない | 技術スタックの選定、予算制約、リリース期限 |
| **中** | 議論は進められるが、精度が落ちる。仮定で進めるリスクがある | ユーザーの利用頻度、好みのUI方向性 |
| **低** | あると議論の質が上がるが、なくても合理的な判断ができる | 参考にしたいサービス、過去の失敗事例 |

## 質問すべき場合

- ユーザーの好み・意向に関すること（デザインの方向性、優先度の感覚）
- ビジネス上の判断が必要なこと（KPI、リリーススケジュール）
- エージェント間の対立が解消できず、ユーザー判断が必要な場合
- ユーザーの実際の利用シーン・行動に関すること
- 前提条件の確認（技術スタック、外部サービスの利用可否）

## 質問すべきでない場合

以下はユーザーに聞かず、エージェント間で解決する：

- コードベースを調査すれば分かること → Engineer に確認を依頼
- ペルソナ情報から推測できること → UX Analyst に確認を依頼
- 一般的なベストプラクティスで回答できること → 自分で回答
- エージェント間の議論で解決できること → 関連エージェントと議論

## 各エージェント固有の質問例

| エージェント | 質問すべき例 |
|-------------|-------------|
| **Product Manager** | ビジネスKPI・目標数値、競合状況、リリーススケジュール制約、ユーザーからの直接フィードバック |
| **UX Analyst** | ペルソナの追加情報（実際の利用シーン、頻度）、ユーザーテストの有無、既存ユーザーの不満・要望、アクセシビリティ要件 |
| **Engineer** | パフォーマンス要件（想定データ量、同時接続数）、既存の技術的負債、サードパーティサービスの利用可否、モバイル/デスクトップの対応範囲 |
| **Designer** | デザインの好み・トーン、既存UIで気に入っている部分/不満な部分、参考にしたいアプリ、PC/スマホの利用比率 |
| **Behavioral Psychologist** | ユーザーの既存の習慣やルーティン、機能を使い始めるきっかけの想定、過去に使われなかった機能の事例、ユーザーの意思決定スタイル |

## User Liaison からの応答

### 質問が差し戻された場合

```
[ASK_USER_REJECTED]
理由: [なぜユーザーに聞かなくてよいか]
代替アクション: [コードを調査する / ペルソナを再確認する / 〇〇と議論する]
```

→ 差し戻された場合は、指示された代替アクションを実行する。

### ユーザー回答が届いた場合

```
[USER_RESPONSE]
質問: [元の質問の要約]
回答: [ユーザーの回答]
影響: [この回答が議論に与える影響]
アクション: [各エージェントが取るべき次のアクション]
```

→ 回答を自分の分析に反映し、必要に応じて議論を再開する。
