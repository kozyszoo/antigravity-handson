# 各ステップの詳細

各ステップの担当エージェント、フロー、完了条件、出力先を定義する。

---

## Step 1: 課題・目的の深掘り

**担当**: PM（リード）+ UX Analyst + Behavioral Psychologist
**目的**: 機能の本質的な価値を明確化

**フロー**:
1. Facilitator がユーザーにヒアリング（課題・目的・動機）
2. PM が課題を構造化し、ビジネス価値を評価
3. UX Analyst がペルソナ視点で「本当にこのペルソナが求めているか？」を批判的に評価
4. **Behavioral Psychologist** がフォッグ行動モデルで評価:
   - 「ユーザーはこの課題を解決する動機が十分か？」
   - 「解決行動を取る能力/容易さはあるか？」
   - 「行動を起こすきっかけは何か？」
5. 3者の意見が対立する場合、全てをユーザーに提示

**PMへの問い**:
- この課題の優先度は？ビジネスインパクトは？
- 解決しないとどうなる？

**UX Analystへの問い**:
- ペルソナ[名前]は本当にこの課題で困っている？
- ペルソナの行動パターンと矛盾しないか？

**Behavioral Psychologistへの問い**:
- ユーザーは「困っている」と言っているが、実際に行動を変えるほど困っているか？
- この課題の解決は習慣化に繋がるか、一回きりか？
- 現状維持バイアスを超えるだけの動機があるか？

**ユーザーに確認すべき事項**:
- この課題に至った背景・きっかけ（なぜ今これが必要だと思ったか）
- 既存の運用やワークアラウンドの有無（今どう対処しているか）
- 想定している利用シーン（いつ・どこで・どんな状況で使うか）
- 優先度の感覚（他の課題と比べてどのくらい重要か）

**完了条件**: 課題と目的が明確化され、ペルソナとの整合性 + 行動実現性が確認された
**出力先**: `1_purpose.md`

## Step 2: 代替案・既存機能の検討

**担当**: 全員参加（Engineer: コード調査、Designer: UI案）
**目的**: 新規開発の必要性を検証し、代替案を比較

**フロー**:
1. **Engineer** がコードベースを調査（`Glob`, `Grep`, `Read`で既存実装を把握）
2. **競合リサーチ** を並列で実行（`references/external_research.md` 参照）
3. **Designer** がUI観点から3つ以上の代替案を提示（ASCII artで視覚化）
4. **PM** がビジネス観点で各案を評価
5. **UX Analyst** がペルソナ観点で各案を評価
6. **Behavioral Psychologist** が各案の行動実現性を評価:
   - 「案Aは習慣ループが成立するか？」
   - 「案Bは選択肢が多すぎて選択麻痺を起こさないか？」
   - 「案Cのデフォルト設計は適切か？」
7. **Engineer** がDesignerの案に対して技術的実現性・リスクを反論
8. **Designer** がEngineerの制約に対して代替UIを提案
9. Facilitator が議論を整理してユーザーに提示

**代替案提示フォーマット**:
```
チームで3つの代替案を議論しました：

案A: [概要]
  PM: 工数 [小/中/大]、ビジネス価値 [低/中/高]
  Engineer: 技術的難易度 [低/中/高]、リスク: [...]
  Designer: UIイメージ [ASCII art]
  UX Analyst: ペルソナ[名前]の評価 [...]
  Behavioral Psychologist: 行動実現性 [高/中/低]、懸念: [...]

案B: ...
案C: ...

⚡ チーム内の対立点:
  - [Engineer vs Designer: ...]
  - [PM vs UX Analyst: ...]

推奨: 案[X]（理由: ...）
どれが最も適切だと思いますか？
```

**ユーザーに確認すべき事項**:
- 参考にしているアプリやサービスがあるか
- 既存UIで気に入っている部分・不満な部分
- PC/スマホどちらを主に使うか（利用比率）
- 各案に対する直感的な好み・違和感

**完了条件**: 代替案を3つ以上提示し、チーム議論を経て推奨案を選定
**出力先**: `2_alternatives.md`

## Step 3: スコープ決定（MVP定義）

**担当**: PM（リード）+ Engineer + UX Analyst + Behavioral Psychologist
**目的**: 最小限の実装範囲を決定

**フロー**:
1. **PM** がMVP・Nice to Have・Futureに分類
2. **Engineer** が各項目の技術的工数を見積もり、PMの分類に反論
   - 「PMはMVPと言っているが、技術的に複雑すぎる。Nice to Haveに落とすべき」
   - 「PMはFutureと言っているが、今やらないと後で大幅な改修が必要」
3. **UX Analyst** がペルソナ視点で優先度に反論
   - 「PMはNice to Haveと言っているが、ペルソナ[名前]にとっては必須」
   - 「PMはMVPと言っているが、ペルソナの利用頻度を考えるとFutureでいい」
4. **Behavioral Psychologist** が習慣形成の観点でスコープに反論
   - 「MVPに報酬フェーズが含まれていない。これでは習慣化しない」
   - 「オンボーディングがFutureに分類されているが、初回体験なしに継続利用はありえない」
   - 「フォッグ行動モデルの3要素（動機・能力・きっかけ）がMVPに揃っているかチェック」
5. Facilitator が合意点・対立点を整理してユーザーに提示

**優先度マトリクス**:
```
🔴 MVP（チーム合意）
  - [全員が必須と認めた項目]

🟡 議論中（対立あり）
  - [項目]: PM=[MVP] / Engineer=[Nice to Have]（理由: ...）
  - [項目]: UX=[MVP] / PM=[Future]（理由: ...）

🟢 Future（チーム合意）
  - [全員がFutureで合意した項目]

対立している項目について、どう判断しますか？
```

**ユーザーに確認すべき事項**:
- リリーススケジュールの制約（いつまでに出したいか）
- 開発リソースの制約（自分一人か、チームか）
- MVP に絶対含めたい/含めなくてよいと思う項目
- エージェント間の対立点に対するユーザーの判断

**完了条件**: MVP・Nice to Have・Futureに分類完了（対立はユーザー判断で解決）
**出力先**: `3_scope.md`

## Step 4: 要件定義ドキュメント生成

**担当**: UX Analyst（リード）+ PM + Engineer + Behavioral Psychologist（レビュー）
**目的**: ユーザー視点での要件定義書を作成

**フロー**:
1. **UX Analyst** がペルソナベースのユーザーストーリーを作成
2. **PM** が受け入れ条件を定義
3. **Engineer** が技術的実現性の観点からレビュー・反論
   - 「この受け入れ条件は現在のアーキテクチャでは実現困難」
   - 「この操作は既存のAPIでは対応できない」
4. **Behavioral Psychologist** が動機付けの観点からレビュー
   - 「ユーザーストーリーの "So that" は本当に内発的動機か？」
   - 「受け入れ条件に "ユーザーが迷わない" ための要素が含まれているか？」
   - 「初回利用→習慣化のトリガーが要件に含まれているか？」
5. 全員の指摘を受けて修正
6. Facilitator が最終版をユーザーに提示

**出力内容**:
1. 機能概要
2. ユーザーストーリー（As a [ペルソナ名] / I want / So that）
3. 受け入れ条件（Given / When / Then）
4. 行動設計メモ（Behavioral Psychologistによる注意点）
5. チーム内で議論になった点と解決策

**ユーザーに確認すべき事項**:
- ユーザーストーリーの「So that（得られる価値）」がユーザーの意図と合っているか
- 受け入れ条件に漏れがないか（エッジケース、エラー時の振る舞い）
- 非機能要件（パフォーマンス、データ量、同時アクセス）

**完了条件**: ユーザーストーリーと受け入れ条件が全エージェントの合意を得た
**出力先**: `4_requirements.md`

## Step 5: UI設計プロンプト生成

**担当**: Designer（リード）+ UX Analyst + Behavioral Psychologist + 全員レビュー
**目的**: UI設計のためのプロンプトを生成

**フロー**:
1. **Designer** が画面レイアウト・インタラクションを提案（ASCII art）
2. **UX Analyst** がペルソナの操作習慣・認知負荷の観点から批判
3. **Engineer** が技術的制約から批判（既存コンポーネントとの整合性）
4. **Behavioral Psychologist** が行動設計の観点から批判:
   - 「デフォルト値が設定されていない → 選択麻痺のリスク」
   - 「初回操作のフィードバックが弱い → 習慣ループの報酬が不足」
   - 「CTAが目立たない → きっかけ（Prompt）が弱い」
   - 「情報量が多すぎる → 認知負荷がワーキングメモリの限界を超える」
5. **Designer** が批判を受けて修正案を提示
6. Facilitator が最終版をユーザーに提示

**出力内容**（各画面について）:
1. 画面の目的
2. レイアウト図（ASCII art）
3. 主要コンポーネント（既存の再利用を優先）
4. ユーザーフロー
5. チーム議論で変更になった点
6. 実装プロンプト

**ユーザーに確認すべき事項**:
- UI案に対する好み・違和感
- 操作フローが自分の想定と合っているか
- デザインのトーン（シンプル志向 vs リッチ志向）

**完了条件**: 画面ごとのプロンプトが全エージェントのレビューを通過
**出力先**: `5_ui_prompt.md`
