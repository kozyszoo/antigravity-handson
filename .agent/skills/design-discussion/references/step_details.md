## Step 1: 課題・目的の明確化

**担当**: solution-architect（リード）+ engineer + product-manager + user-liaison

**目的**: 「何を解決したいのか」「制約条件は何か」を確定し、全員が同じ前提で議論できる状態にする

### フロー

```
1a. solution-architect が設計課題を構造化
    - 「何を決定する必要があるか」を明確にする
    - engineer に「既存コードを調査して」と依頼
    - product-manager に「ビジネスコンテキストを整理して」と依頼

1b. engineer が並列でコードベース調査
    - 関連ファイル・既存実装を Glob/Grep/Read で調査
    - 技術的制約条件を特定

1c. product-manager が並列でビジネスコンテキストを整理
    - 解決すべきビジネス課題を定義
    - 期限・コスト・優先度の制約を確認

1d. user-liaison が確認事項を判定
    - コードから分からない前提をユーザーに確認
    - 議論開始前の重要な制約をクリアにする

1e. solution-architect が結果を統合して broadcast
    - 確定した課題・制約・成功基準を全員に共有
    - 「この理解で合ってますか？」とユーザーに確認
```

### 明確化すべき事項チェックリスト

- [ ] 解決したい技術的問題は何か
- [ ] 変更できない制約条件は何か（技術スタック、期限、予算）
- [ ] 成功基準は何か（パフォーマンス指標、可用性要件 etc.）
- [ ] 影響を受ける既存コンポーネントは何か
- [ ] 「設計しない」場合のリスクは何か

### 完了条件

全員が以下に合意していること:
- 技術的問題の定義
- 制約条件の一覧
- 成功基準

### 出力: 1_clarification.md

`references/output_templates.md` の `1_clarification.md` テンプレートを使用。

## Step 2: 複数の設計案を提示（2-4案）

**担当**: solution-architect（リード）+ engineer + product-manager

**目的**: 少なくとも2案、最大4案の実装アプローチを提案し、初期評価を行う

### フロー

```
2a. solution-architect が 2-4 の設計案を生成
    - 「現状維持」案を必ず含める
    - 各案を ASCII art や構造図で可視化
    - broadcast で全員に共有

2b. engineer が並列で各案を技術評価
    - 技術的実現性（低/中/高）
    - 工数の3点見積もり（楽観/最頻/悲観）
    - 既存コードとの整合性

2c. product-manager が並列で各案をビジネス評価
    - ビジネス価値スコア
    - スコープとの適合性
    - 「今やるべきか」の判定

2d. solution-architect が初期スコアリング表を作成
    - 5軸で各案を評価
    - engineer/PM の評価を反映
```

### 設計案の条件

- 最低 2 案、最大 4 案（多すぎると議論が散漫になる）
- 必ず「現状維持」案を含める
- 各案に具体的な実装アプローチ（「〇〇ライブラリを使う」レベルの具体性）
- ASCII art や構造図で可視化する

### 完了条件

- [ ] 2-4 の設計案が生成されている
- [ ] 各案に技術的実現性・工数評価がある
- [ ] 各案にビジネス価値評価がある
- [ ] 初期スコアリング表が完成している

## Step 3: 厚めの比較・深掘り

**担当**: 全員 + devils-advocate（R2 で弱点を突く）

**目的**: メリデメ/リスク/工数/影響範囲/拡張性の多面評価を行い、最優位案を絞り込む

### フロー

```
3a. 詳細技術比較（engineer リード）
    - パフォーマンス・スケーラビリティ・セキュリティの軸で比較
    - 移行コスト・後戻りコストを評価

3b. Devils Advocate セッション
    - devils-advocate が全案を批判
    - 致命的リスク・見落とし・前提の誤りを指摘
    - 他エージェントが反論または認める

3c. 批判を受けた修正
    - 致命的指摘は案を修正または別案を追加
    - 重要指摘はリスクとして記録
    - 過剰指摘は却下記録

3d. 最終詳細比較表の作成（solution-architect）
    - 全軸の最終スコア
    - Devils Advocate の評価を反映
    - 対立点を明示

3e. ユーザーへの提示
    - 比較表を提示
    - 対立点でユーザー判断が必要な場合は質問
```

### 比較軸（7軸）

| 評価軸 | 説明 | 主な評価者 |
|--------|------|----------|
| 技術的実現性 | 実装の容易さ・リスクの低さ | engineer |
| 工数・コスト | 実装工数（楽観/最頻/悲観） | engineer |
| 保守性・拡張性 | 将来の変更容易性 | solution-architect |
| ビジネス価値 | ユーザー・事業への影響度 | product-manager |
| リスク | 既知・未知リスクの総合 | devils-advocate |
| 後戻りコスト | ロールバックの難しさ | engineer |
| 運用複雑さ | デプロイ・監視・保守の負担 | engineer |

### 完了条件

- [ ] 全案が Devils Advocate に批判されている
- [ ] 各批判に応答または修正がある
- [ ] 詳細比較表が完成している
- [ ] 未解決の対立点が明示されている

### 出力: 3_comparison.md

`references/output_templates.md` の `3_comparison.md` テンプレートを使用。

## Step 4: 推奨案の決定と次のアクション

**担当**: solution-architect + product-manager + engineer + user-liaison（承認）

**目的**: 推奨案を確定し、ADR とアクションリストを作成、Plan ファイルへの反映指示を出す

### フロー

```
4a. solution-architect が推奨案を決定
    - スコアリングと議論の結果を統合
    - 推奨案の選定理由を明文化
    - 却下した案と理由を記録

4b. ADR（Architecture Decision Record）の作成
    - コンテキスト・決定・根拠・影響を記述
    - 残留リスクと対策を記録

4c. アクションリストの作成
    - 実装に必要な具体的アクション
    - 優先度・担当・期限を設定

4d. user-liaison がユーザーに提示
    - 推奨案・理由・却下理由を説明
    - アクションリストを提示
    - 「承認しますか？」と確認

4e. Plan ファイル反映指示（承認後）
    - ADR の要点を Plan に追記するための指示を生成
    - 実装上の注意点を Plan に含める
```

### 完了条件

- [ ] 推奨案と選定理由が明文化されている
- [ ] 却下した案と理由が明文化されている
- [ ] ADR が作成されている
- [ ] アクションリストが作成されている
- [ ] ユーザーが承認している

### Plan ファイル反映

Step 4 完了後、以下の内容を Plan ファイルに追記することを推奨:

```markdown
### 設計決定

採用アーキテクチャ: [案名]
決定根拠: [主な理由]
ADR: .claude/design_discussion/sessions/<slug>/4_decision.md

実装の前提:
- [前提1]
- [前提2]

注意点:
- [Devils Advocate 指摘で残存するリスク]
```

### 出力: 4_decision.md

`references/output_templates.md` の `4_decision.md` テンプレートを使用。
